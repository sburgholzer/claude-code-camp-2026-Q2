# 01 · Struct Skeleton (Python port)

Behavior port of `ruby/01_struct_skeleton` — defines the three data
structures the rest of the port passes around: `Tool`, `Message`, and
`Context`. See `../README.md` for the one-time venv setup this step
depends on.

Ruby represents these as `Struct`s for a lightweight, readable value
object. The closest Python stdlib equivalent is the `dataclasses`
module, used here for `Tool` and `Message`. `Context` is a plain class,
matching the Ruby version.

## Data Structures

### `boukensha.tool.Tool`
| Field | Description |
|---|---|
| `name` | The name of the tool |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | The arguments that need to be passed in |
| `block` | The actual code that runs when the tool is called |

**Examples:**
```
#<Tool name=move description=Move the player in a direction (north, so params=['direction']>
```

### `boukensha.message.Message`
A single unit of conversation between the user and the agent.

| Field | Description |
|---|---|
| `role` | Who is speaking — `"user"`, `"assistant"`, or `"tool_result"` |
| `content` | The text generated by the agent or provided by the user |
| `tool_use_id` | Links a tool result back to the specific tool call that requested it |

**Examples:**
```
#<Message role=user content=Explore north and tell me what you find....>
#<Message role=assistant content=Sure, let me head north and take a look....>
```

### `boukensha.context.Context`
Holds everything BOUKENSHA needs to make an API call.

| Field | Description |
|---|---|
| `task` | The task class (e.g. `Player`) this context is being built for |
| `system` | The resolved system prompt, sent as a separate field, never inside `messages` |
| `messages` | The full conversation history, replayed on every turn |
| `tools` | The registered tools the agent is allowed to invoke |

```
#<Context task=player turns=2 tools=1>
```

## Porting notes

- Ruby symbols (`:direction`, `:user`) have no Python equivalent —
  plain strings are used throughout. This means `Tool`'s printed
  `params=` list renders as `['direction']` rather than Ruby's
  `[:direction]`; everything else matches.
- Ruby's inclusive-range truncation (`str[0..40]`, `str[0..60]`) is
  41/61 characters; the Python equivalents are the slices `[:41]` and
  `[:61]`.
- `Context.__init__` takes `task` and `system` as keyword-only
  arguments (`*, task, system=None`), mirroring Ruby's forced-keyword
  `initialize(task:, system: nil)`.
