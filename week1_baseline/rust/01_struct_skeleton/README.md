# 01 · Struct Skeleton (Rust port)

Behavior port of `ruby/01_struct_skeleton` / `python/01_struct_skeleton` —
defines the three data structures the rest of the port passes around:
`Tool`, `Message`, and `Context`. See `../README.md` for the one-time
workspace setup this step depends on.

Ruby represents these as `Struct`s, Python as `dataclasses`. Rust has
neither, so `Tool` and `Message` are plain structs with a hand-written
`Display` impl doing the same field-truncation formatting, and `Context` is
a plain struct too (matching Ruby/Python, which use a plain class for it).

## Data Structures

### `boukensha_01_struct_skeleton::tool::Tool`
| Field | Description |
|---|---|
| `name` | The name of the tool |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | The arguments that need to be passed in |
| `block` | The actual code that runs when the tool is called |

**Examples:**
```
#<Tool name=move description=Move the player in a direction (north, so params=["direction"]>
```

### `boukensha_01_struct_skeleton::message::Message`
A single unit of conversation between the user and the agent.

| Field | Description |
|---|---|
| `role` | Who is speaking — `"user"`, `"assistant"`, or `"tool_result"` |
| `content` | The text generated by the agent or provided by the user |
| `tool_use_id` | Links a tool result back to the specific tool call that requested it |

**Examples:**
```
#<Message role=user content=Explore north and tell me what you find....>
#<Message role=assistant content=Sure, let me head north and take a look....>
```

### `boukensha_01_struct_skeleton::context::Context`
Holds everything BOUKENSHA needs to make an API call.

| Field | Description |
|---|---|
| `task` | The task type (e.g. `Player`) this context is being built for |
| `system` | The resolved system prompt, sent as a separate field, never inside `messages` |
| `messages` | The full conversation history, replayed on every turn |
| `tools` | The registered tools the agent is allowed to invoke |

```
#<Context task=player turns=2 tools=1>
```

## Porting notes

- Ruby symbols (`:direction`) and Python's single-quoted list repr
  (`['direction']`) have no Rust equivalent — `Tool`'s printed `params=`
  list is Rust's `Vec<&str>` `Debug` format, `["direction"]`
  (double-quoted). Same category of key-representation divergence Python's
  port already flagged for Ruby symbols; everything else matches.
- Ruby's inclusive-range truncation (`str[0..40]`, `str[0..60]`) is
  41/61 characters; the Rust equivalents are
  `description.chars().take(41)` and `content.chars().take(61)` —
  character-based (not byte-slicing) for the same reason
  `rust/00_config`'s example already truncates the system prompt via
  `.chars().take(60)`: byte slicing panics on a non-ASCII boundary.
- **No `PROMPTS_DIR`, no shipped `prompts/` dir this step** — matches
  Python's `config.py`, which drops the `PROMPTS_DIR` class attribute that
  `00_config` has. `examples/example.rs` calls `Player::system_prompt` with
  only `user_prompts_dir` (`default_prompts_dir: None`), relying entirely
  on the shared `.boukensha/prompts/player/system.md` override fixture.
- **`Context<T: Task>` uses a compile-time type parameter, not a runtime
  value, for `task`.** Ruby/Python store the literal task class as a
  runtime field and call `.task_name()` on it; Rust's `Task` trait already
  makes every method an associated function with no instance to hold, so
  there's nothing to store at runtime. `Context<T>` carries a
  zero-sized `PhantomData<T>` instead, and reads `T::task_name()` — the
  task is fixed at the type level (`Context::<Player>::new(...)`) rather
  than passed as a constructor argument.
- **`Tool.block` is `Box<dyn Fn(&str) -> String>`**, chosen to match the
  one closure literal that actually exists in this step's example
  (`|direction| format!("You move {direction}...")`) — boxing is required
  because `Context.tools` is a `HashMap<String, Tool>` holding
  heterogeneous closures, which Rust can only store uniformly via type
  erasure (Ruby/Python don't need this since any lambda/callable fits in
  one hash/dict). The block is never invoked in this step's example, only
  constructed and displayed, so this signature is expected to generalize
  once a later step actually dispatches tool calls.

## Run Example

```bash
./week1_baseline/bin/rust/01_struct_skeleton
```

Expected output (values from your `.boukensha/`):

```
=== Boukensha Step 1: Struct Skeleton ===

Config:   #<Boukensha::Config dir=/home/andrew/Sites/Claude-Code-Camp/.boukensha tasks=player>
Context:  #<Context task=player turns=2 tools=1>
Tool:     #<Tool name=move description=Move the player in a direction (north, so params=["direction"]>
Messages:
  #<Message role=user content=Explore north and tell me what you find....>
  #<Message role=assistant content=Sure, let me head north and take a look....>
```
